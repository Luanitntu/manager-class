# Phase 05 - UI Design Contract

> Visual and interaction contract for Phase 5. Generated by gsd-ui-researcher for the planner, executor, checker, and auditor.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind 4 via `@tailwindcss/vite`; no shadcn; no Vuetify; no SCSS |
| Preset | Not applicable |
| Component library | Local Vue 3/Nuxt `Ui*` primitives from `client/app/components/ui`; `App*` infrastructure wrappers where already used |
| Icon library | `@mdi/font` through `AppIcon`; do not introduce another icon library |
| Font | Nunito via `--st-font-family`; fallback `Arial, sans-serif` |
| Token source | Existing `--st-*` tokens in `client/app/assets/css/main.css` |
| Component prefix | `Ui*` for design-system primitives; `App*` remains app infrastructure |

Source: `PROJECT.md`, `ROADMAP.md`, `REQUIREMENTS.md`, prior Phase 2-4 UI-SPECs, `client/app/components/ui/README.md`, and current source scan.

## Phase Boundary

Phase 5 proves the v1.1 Tailwind migration is complete, stable, and free of accidental Vuetify/SCSS regressions. It is a verification and cleanup phase, not a redesign phase.

In scope:

- Run and record frontend lint, typecheck, and build evidence from `client/`.
- Run static scans for Vuetify tags, SCSS/style markers, removed dependency surface, and UI-kit data-boundary violations.
- Smoke-check the app shell, calendar-first teacher workflow, and Phase 4 priority pages on desktop and mobile-width viewports.
- Capture screenshot or manual QA evidence for the verification targets.
- Remove stale low-risk Phase 5 cleanup files only when removal does not alter active teacher/student behavior.
- Document intentionally deferred non-priority/admin/general page UI debt with exact file scope and rationale.
- Apply tiny UI polish only when a verification check finds overflow, overlap, clipping, unreadable text, broken wrapping, or missing state in a v1.1 smoke target.

Out of scope:

- New product features or new workflows.
- Broad whole-codebase beauty pass.
- Admin/center page redesigns.
- Center role work.
- Backend changes unless required by an actual verification failure.
- New UI dependencies, shadcn, Radix, Headless UI, DaisyUI, Flowbite, or third-party registries.

## Tailwind-Only Verification Contract

All Phase 5 edits must preserve the Tailwind-first shared UI direction from Phases 2-4.

Required old-to-new expectations for any touched active UI:

| Old pattern | Required treatment |
|-------------|--------------------|
| `v-*` components in smoke targets | Remove or replace with existing `Ui*` primitives and `AppIcon` |
| `<style>`, `lang="scss"`, `.scss`, `:deep(.v-)` in smoke targets | Remove or replace with Tailwind utilities/tokenized primitives |
| Stale style files not imported by active UI | Delete only after proving they are unused |
| Admin/center/general old UI outside v1.1 priority scope | Document as deferred; do not redesign in Phase 5 |
| UI-kit feature data imports | Remove from `client/app/components/ui`; feature ownership stays in pages/composables/feature components |

Data ownership rule:

- `Ui*` components remain data-agnostic and must not import feature composables, auth stores, API helpers, `$fetch`, tenant-aware logic, or route state.
- Pages and feature components keep validation, mutation, filtering, tenant-scoped data, formatting, and smoke-state ownership.

## Spacing Scale

Declared values for Phase 5 smoke checks and any tiny polish edits. All values are multiples of 4.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, badge gaps, dense metadata |
| sm | 8px | `gap-2`, `p-2` | Compact row internals, toolbar action gaps |
| md | 16px | `gap-4`, `p-4` | Default cards, lists, forms, table cells |
| lg | 24px | `gap-6`, `p-6` | Page sections, dialog/card padding |
| xl | 32px | `gap-8`, `p-8` | Page header separation and major route groups |
| 2xl | 48px | `gap-12`, `py-12` | Empty canvases only |
| 3xl | 64px | `gap-16`, `py-16` | Not used in Phase 5 operational polish |

Exceptions:

- Minimum interactive target is 44px for shell nav, icon buttons, primary buttons, row actions, calendar cells that create sessions, dialog controls, and pagination.
- Compact 36px controls are allowed only for dense filters, pagination, and calendar previous/next controls.
- Shell topbar remains 64px; drawer remains 256px on desktop.
- Calendar board keeps intentional horizontal scroll via `min-w-[760px]`; no other smoke target may create page-level horizontal scroll.
- Dialog viewport bound remains `width: min(100vw - 32px, maxWidth)` with internal body scroll.
- Cards use `--st-radius` 8px. Do not put cards inside cards.

## Typography

Touched Phase 5 UI and verification criteria use exactly these four sizes and two weights.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 14px | 600 | 1.5 | Labels, table headers, metadata, badge/button text |
| Body | 16px | 400 | 1.5 | Default row text, inputs, table cells, page copy |
| Heading | 20px | 600 | 1.2 | Card headings, dialog titles, section titles, metric values |
| Page Title | 24px | 600 | 1.2 | Route page titles only |

Rules:

- Use weight 400 for content and 600 for emphasis/actions. Do not introduce weights 500, 700, or 800 in touched Phase 5 UI.
- Letter spacing is `0`.
- Do not scale font size with viewport width.
- Long names, emails, class names, action strings, entity IDs, and route labels must use `min-w-0`, wrapping, truncation, or contained table overflow.
- Table body text may use 14px when using the existing `UiTable` dense contract; page body/form/list text remains 16px.

## Color

Use existing `--st-*` tokens as source of truth.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--st-bg-soft` `#f5f8ff` | App/page background, empty canvas, soft bands |
| Secondary (30%) | `--st-surface` `#ffffff` | Cards, dialogs, tables, inputs, drawer, topbar |
| Accent (10%) | `--st-accent` `#ff6b00` | Reserved elements listed below only |
| Destructive | `#ef4444` | Error alerts and destructive confirmations only |

Accent reserved for:

- Calendar-first emphasis such as current/urgent session markers.
- Non-destructive warning/attention badges.
- Existing profile/account accent details already shipped in Phase 4.
- Auth/brand continuity already present in active surfaces.

Do not use accent for default primary actions. Primary actions use `--st-primary` `#0d6efd`; hover/pressed uses `--st-primary-dark` `#075dee`.

Additional semantic colors:

| Token | Value | Usage |
|-------|-------|-------|
| Primary | `--st-primary` `#0d6efd` | Main CTAs, selected nav state, links, selected tabs |
| Primary Dark | `--st-primary-dark` `#075dee` | Primary hover/active |
| Secondary Brand | `--st-secondary` `#49beff` | Informational badges and calm highlights |
| Text | `--st-text` `#17233c` | Main text |
| Muted | `--st-muted` `#475569` | Secondary text, helper text, metadata |
| Border | `--st-border` `#dce5f1` | Borders, dividers, inputs, table/list separators |
| Focus | `--st-focus` | Focus ring via existing Tailwind ring equivalents |
| Success | `#10b981` | Passed checks, successful saves, completed states |
| Warning | `#f97316` | Non-destructive warnings and attention |
| Error | `#ef4444` | Errors and destructive state |

Color rules:

- Status cannot be color-only; badges and evidence summaries must include text.
- Disabled state uses opacity plus disabled semantics.
- Focus-visible state remains visible on every keyboard-focusable control.
- Do not shift the app into a one-hue blue palette; keep neutral surfaces, blue primary, orange attention, green success, red error.

## Interaction Rules

Keyboard and focus:

- Every input, select, button, row button/link, tab, dialog close, pagination control, calendar cell, event button, menu trigger, and confirm action is keyboard reachable.
- Focus-visible ring uses `--st-focus` or existing `focus-visible:ring-2 focus-visible:ring-blue-100`.
- Dialog opens move focus into the dialog and restore focus to the opener when closed.
- Escape closes non-destructive dialogs and menus. Destructive confirmations require explicit cancel/confirm.
- Icon-only controls require `aria-label`.

Responsive behavior:

- Verify desktop width and mobile width for every smoke target.
- Toolbars wrap instead of overflowing.
- Tables use horizontal overflow wrappers on mobile; no page-level horizontal scroll except the calendar board's contained grid.
- Dialogs fit the viewport and keep footer actions reachable.
- Mobile drawer and profile dialog do not create horizontal scroll.

State behavior:

- Every smoke target must show or document loading, empty, error, success, and populated states as applicable.
- Loading states preserve approximate layout dimensions with `AppSkeleton`/`UiSkeleton`.
- Mutating controls show loading and disabled state.
- Error states include problem text and a retry/next-step path when the source flow supports it.
- Pagination range math and page-size behavior remain compatible with `TablePager`/`UiPagination`.

Accessibility:

- Inputs/selects have visible labels or accessible labels.
- Tables have captions through `UiTable` or equivalent screen-reader text.
- Status badges include text.
- Alerts/toasts use semantic roles from `UiAlert`/`UiToast`.
- Color is never the only signal for status, selection, validation, active nav, today, drop target, or destructive action.

## Copywriting Contract

Preserve existing app/i18n copy. Phase 5 should only add or change copy for verification-driven tiny polish, evidence docs, or cleanup notes.

| Element | Copy |
|---------|------|
| Primary CTA | Preserve shipped labels: `Create session`, `Save session`, `Create assistant`, `Save salary settings`, `Save profile`, `Clear filters` |
| Empty state heading | Preserve shipped headings: `No assistants found`, `No class assignments yet`, `No sessions scheduled`, `No salary records yet`, `No salary history yet`, `No audit logs found`, `No upcoming sessions`, `No classes this week` |
| Empty state body | Preserve shipped bodies: `Create an assistant or change your search.`, `Assign this assistant from a class.`, `Adjust filters to view matching activity.`, and route-specific existing copy |
| Error state | Use extracted API error when available; fallback `Something went wrong. Check the details and try again.` |
| Success state | Preserve shipped success copy such as `Profile updated.` and existing toast/i18n success messages |
| Destructive confirmation | Preserve noun-specific confirmations; session delete remains `This action cannot be undone. Delete this session?` or current localized equivalent |

Copy rules:

- CTA labels use specific verb + noun. Do not replace object-specific labels with generic `Save`, `Create`, or `Submit`.
- Keep Vietnamese/i18n labels where current surfaces already use them.
- Do not add visible instructional text explaining UI mechanics, screenshots, QA process, or keyboard shortcuts inside the app.
- Evidence documents may use concise technical language; app UI copy stays product-focused.

Destructive actions in Phase 5:

- Do not introduce new destructive product actions.
- File cleanup is not product UI; document removed stale files and proof that active surfaces are unaffected.
