# Phase 02 - UI Design Contract

> Visual and interaction contract for Phase 2. Generated by gsd-ui-researcher for the planner, executor, checker, and auditor.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind 4 via `@tailwindcss/vite`; no shadcn; no Vuetify; no SCSS |
| Preset | Not applicable |
| Component library | Local Vue 3/Nuxt shared components only |
| Icon library | `@mdi/font` through `AppIcon`; do not introduce another icon library |
| Font | Nunito via `--st-font-family`; fallback `Arial, sans-serif` |
| Token source | Existing `--st-*` tokens in `client/app/assets/css/main.css` |
| Component prefix | `Ui*` for design-system primitives; `App*` remains app infrastructure |

Source: `02-CONTEXT.md` D-01 through D-21, `REQUIREMENTS.md` UIKIT-00 through UIKIT-05, `client/app/assets/css/main.css`, `AppIcon.vue`.

## Phase Boundary

Phase 2 must define and prove a Tailwind-only UI kit. It must not become a broad page redesign or high-risk workflow migration.

In scope:

- Document token, variant, naming, and usage rules for downstream phases.
- Add shared `Ui*` components for layout, controls, forms, data display, feedback, and overlay foundation.
- Migrate only small proof surfaces where parity is practical: `AppSkeleton`, `TablePager`, `AppToast`, `ClassLocation`, or similarly small shared surfaces.
- Use high-risk dialogs (`SessionDialog`, `StudentDetailDialog`, `AssistantDetailDialog`) as requirements input for dialog/form/table needs, not default full migration targets.

Out of scope:

- Full priority page redesign for `/assistants`, `/assistants/[id]`, `/audit-logs`, `/profile`.
- Full app shell migration.
- Full migration of `SessionDialog`, `StudentDetailDialog`, or `AssistantDetailDialog` unless the planner explicitly splits a low-risk foundation-only slice.
- New product features, backend changes, Vuetify bridges, SCSS, scoped CSS, CSS modules, page CSS, component CSS, or broad helper classes.

## Tailwind-Only Contract

All new or touched Phase 2 UI must be styled with Tailwind utility classes and existing CSS variable tokens. New CSS is allowed only in `client/app/assets/css/main.css` for Tailwind entry, font import, `:root` tokens, `@theme`, and tiny base reset rules.

Do not add:

- `<style scoped>`, `<style module>`, SCSS, CSS modules, or page/component CSS.
- New global helper classes for component styling.
- Vuetify components, Vuetify-like wrapper APIs, or `v-*` class coupling.
- One-off class recipes repeated across pages when a `Ui*` component is appropriate.

If Tailwind utilities become too complex for a repeated pattern, create or extend a `Ui*` primitive with a small variant surface.

## Spacing Scale

Declared values for new `Ui*` components. All values are multiples of 4.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, badge gaps, tight inline metadata |
| sm | 8px | `gap-2`, `p-2` | Compact control internals, stacked label/hint gaps |
| md | 16px | `gap-4`, `p-4` | Default component padding, list rows, table cells |
| lg | 24px | `gap-6`, `p-6` | Cards, dialogs, page sections |
| xl | 32px | `gap-8`, `p-8` | Page headers, major content groups |
| 2xl | 48px | `gap-12`, `py-12` | Large section separation only |
| 3xl | 64px | `gap-16`, `py-16` | Rare page-level spacing; avoid inside dense tools |

Exceptions:

- Minimum interactive target: 44px for buttons, icon buttons, rows with actions, and dialog close controls.
- Compact controls may use 36px height only for dense table filters, pagination, and secondary toolbar controls.
- `--st-content-max: 1400px` remains the page content max width.
- Existing `--st-space-page-sm: 18px` is legacy compatibility. Do not use it in new `Ui*` components.

Layout rules:

- Cards use 8px radius unless preserving an existing touched surface requires 10px or 12px during a proof migration.
- Do not put cards inside cards. Use full-width sections, lists, tables, or unframed groups inside cards.
- Tables and list rows must keep stable height: 48px minimum for table rows, 56px minimum for list rows with leading media.
- Responsive page containers use `w-full max-w-[var(--st-content-max)] mx-auto px-4 sm:px-6 lg:px-8`.

## Typography

New `Ui*` components use exactly these four sizes and two weights. Existing larger/smaller tokens may remain for untouched legacy parity, but new shared kit APIs must not introduce extra type scales.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 14px | 600 | 1.5 | Form labels, table headers, metadata, button text |
| Body | 16px | 400 | 1.5 | Default content, field text, row text |
| Heading | 20px | 600 | 1.2 | Card headings, dialog titles, section titles |
| Page Title | 24px | 600 | 1.2 | Page headers and major screen titles |

Rules:

- Use weight 400 for content and weight 600 for emphasis/actions. Do not use 500, 700, 800 in new `Ui*` components.
- Letter spacing is `0`.
- Do not scale font size with viewport width.
- Button text must stay on one line only when the label is short; otherwise allow wrapping without changing button height unpredictably.
- Long labels and table cell content must use `min-w-0`, wrapping, truncation, or responsive stacking to prevent overlap.

## Color

Use the existing `--st-*` tokens as the source of truth. Add semantic aliases only if needed for readability; aliases must point to these values unless the planner records a narrow reason.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--st-bg-soft` `#f5f8ff` | App background, empty canvas, soft page bands |
| Secondary (30%) | `--st-surface` `#ffffff` | Cards, dialogs, tables, inputs, nav/sidebar surfaces |
| Accent (10%) | `--st-accent` `#ff6b00` | Reserved elements listed below only |
| Destructive | `#ef4444` | Delete/remove/destructive confirmations only |

Accent reserved for:

- Calendar-first emphasis markers, such as current session/urgent schedule affordance.
- Warning/attention badges that are not destructive.
- Teacher workflow highlights where an item needs immediate action.
- Landing/auth visual continuity already using the orange accent.

Do not use accent for every primary action. Default primary actions use `--st-primary` `#0d6efd`; hover/pressed states may use `--st-primary-dark` `#075dee`.

Additional semantic colors:

| Token | Value | Usage |
|-------|-------|-------|
| Primary | `--st-primary` `#0d6efd` | Main buttons, links, focusable selected states |
| Primary Dark | `--st-primary-dark` `#075dee` | Primary hover/active |
| Secondary Brand | `--st-secondary` `#49beff` | Secondary informational badges and calm highlights |
| Text | `--st-text` `#17233c` | Main text |
| Muted | `--st-muted` `#475569` | Secondary text, helper text |
| Border | `--st-border` `#dce5f1` | Borders, dividers, input outlines |
| Focus | `--st-focus` | Focus ring; implement with Tailwind arbitrary ring where practical |
| Success | `#10b981` | Saved/complete states |
| Warning | `#f97316` | Non-destructive warnings |
| Error | `#ef4444` | Errors and destructive state |

Color rules:

- Text on white uses `--st-text`; secondary text uses `--st-muted`.
- Inputs and table borders use `--st-border`.
- Disabled state uses reduced opacity plus preserved contrast; do not signal disabled by color alone.
- Focus state must be visible on every keyboard-focusable control.
- Do not create a one-hue blue-only UI. Use neutral surfaces, blue primary, orange accent, green success, red destructive.

## Component Inventory

Phase 2 should provide these primitives, with exact names unless implementation finds an existing stronger local convention. Components are slot-first and expose only necessary state props.

### Layout

| Component | Required Contract |
|-----------|-------------------|
| `UiPage` | Constrained page container; default max width `--st-content-max`; responsive padding |
| `UiPageHeader` | Title, eyebrow/subtitle slot, actions slot; no card wrapper |
| `UiToolbar` | Responsive row for filters/actions; wraps cleanly under 640px |
| `UiSection` | Section heading/description/actions slots; unframed by default |
| `UiCard` | Surface with `default`, `soft`, `outlined` variants; 8px radius; no nested-card styling |
| `UiStack` / `UiGrid` | Optional layout helpers only if they reduce repeated Tailwind class bundles |

### Controls and Forms

| Component | Required Contract |
|-----------|-------------------|
| `UiButton` | `variant: primary/secondary/ghost/danger`, `size: sm/md/lg/icon`, `loading`, `disabled`, `to`, leading/trailing icon slots |
| `UiIconButton` | 44px default target, 36px compact, accessible label required |
| `UiInput` | `modelValue`, `type`, `label`, `placeholder`, `hint`, `error`, `disabled`, `required`, leading/trailing slots |
| `UiTextarea` | Same label/error/hint contract; stable min height |
| `UiSelect` | Native select first; `modelValue`, `items`, `label`, `placeholder`, `hint`, `error`, `disabled`, `loading` |
| `UiCheckbox` | Label slot, description/hint, error, disabled |
| `UiSegmentedControl` | Replaces `v-btn-toggle`; keyboard operable; selected state visible |
| `UiActionGroup` | Aligns primary/secondary/destructive actions; stacks on mobile |

### Data Display

| Component | Required Contract |
|-----------|-------------------|
| `UiTable` | Slot-first table wrapper; horizontal overflow container on small screens |
| `UiList` / `UiListItem` | Slot-first rows with leading/trailing/action slots; 56px minimum row |
| `UiBadge` | `tone: neutral/primary/info/success/warning/danger`, `size: sm/md`; optional icon slot |
| `UiAvatar` | Initials or image; deterministic fallback; `size: sm/md/lg` |
| `UiMetricCard` | Label, value, hint/trend, icon slot; no nested cards |
| `UiStatusDot` | Small status marker paired with text; never color-only |
| `UiPagination` | Replaces `TablePager`; page/limit models, range text, previous/next, page size select |
| `UiTabs` | Only if needed by dialog/page foundations; keyboard semantics required |

### Feedback

| Component | Required Contract |
|-----------|-------------------|
| `UiAlert` | `tone: info/success/warning/error`, title slot, default slot, icon slot |
| `UiToast` | Toast item surface for `AppToast`; `role=status` except errors use `role=alert` |
| `UiSkeleton` | Replaces `AppSkeleton` styling; variants `list/table/card/form/detail/calendar` |
| `UiEmptyState` | Icon, heading, body, optional action slot |
| `UiProgress` | Linear progress only where existing surfaces need it |
| `UiSpinner` | Small inline loading indicator for buttons/field adornments |

### Overlay and Dialog Foundation

| Component | Required Contract |
|-----------|-------------------|
| `UiDialog` | `modelValue`, title slot, default slot, footer slot; focus trap; Escape closes unless disabled; backdrop click configurable |
| `UiConfirmDialog` | Title, message, confirm/cancel labels, destructive tone; returns explicit confirm/cancel events |
| `UiModalHeader` / `UiModalFooter` | Optional helpers only if used by multiple dialogs |

Dialog foundation must support later `SessionDialog`, `StudentDetailDialog`, and `AssistantDetailDialog` migrations, but Phase 2 should not fully migrate those workflows by default.

## Component API Rules

- Slot-first by default. Use props only for repeated visual states: `variant`, `size`, `tone`, `loading`, `disabled`, `error`, `hint`, `required`.
- Do not clone Vuetify prop surfaces.
- UI components do not call APIs, stores, or feature composables.
- Validation and business logic stay in pages, composables, or form-layer code.
- Components emit standard Vue events: `update:modelValue`, `click`, `confirm`, `cancel`, `close`.
- Use `AppIcon` inside components for icon rendering. Icon slots should accept either a slot or an MDI name prop only when repeated enough to justify it.
- Any component that renders a native control must forward `id`, `name`, `autocomplete`, `aria-*` where relevant.
- Use `defineProps`/`defineEmits` with explicit TypeScript types.

## Interaction Rules

Keyboard and focus:

- Every control, link, menu trigger, tab, dialog close, pagination button, and destructive confirm action is keyboard reachable.
- Visible focus ring uses `--st-focus` or Tailwind equivalent `focus-visible:ring-2 focus-visible:ring-blue-100 focus-visible:border-blue-500`.
- Dialog opens move focus into the dialog and restore focus to the opener when closed.
- Escape closes non-destructive dialogs. Destructive confirm dialogs may require explicit cancel/confirm.

Responsive behavior:

- Toolbars wrap instead of overflowing.
- Tables use horizontal scroll wrappers below tablet width; key mobile views may use list cards later, but Phase 2 table primitive must not clip.
- Dialog width: `min(100vw - 32px, configured max width)` and max height `calc(100vh - 32px)` with body scroll.
- Toasts anchor bottom-right on desktop and full-width bottom inset on mobile.

States:

- Every control defines default, hover, active, focus-visible, disabled, loading, and error where applicable.
- Every data surface defines loading, empty, error, and populated states.
- Loading state must preserve layout dimensions; use skeletons or reserved min-height instead of layout jumps.

Accessibility:

- Inputs have visible labels or an accessible label.
- Icon-only buttons require `aria-label`.
- Badges/status dots are paired with text.
- Alert/toast roles are semantic.
- Color is never the only signal for status, selection, validation, or destructive action.

## Visual Parity Requirements

Every proof migration in Phase 2 must preserve the current surface behavior and intended layout.

Required parity checks for touched proof surfaces:

- Desktop width and mobile width show no overlap, clipped controls, unreadable text, broken wrapping, or missing states.
- Existing copy and i18n behavior remains unchanged unless this contract explicitly defines replacement copy.
- Pagination range math and page-size options remain compatible with `TablePager`.
- Toast roles, dismiss behavior, stacking, and mobile placement remain compatible with `AppToast`.
- Skeleton variants preserve approximate dimensions for dashboard, table, list, form, detail, and calendar placeholders.
- `ClassLocation` preserves online/offline icon, label, link opening behavior, and inline/chip modes.

If a proof migration cannot preserve parity within small scope, document the blocker and defer that surface to Phase 3/4 instead of widening Phase 2.

## Copywriting Contract

Source copy should keep existing app/i18n strings where already present. New shared components may expose slots so pages supply product copy.

| Element | Copy |
|---------|------|
| Primary CTA | `Save changes` for generic edit forms; feature pages may keep existing i18n labels such as `Save`, `Create assistant`, or localized equivalents |
| Empty state heading | `No data yet` |
| Empty state body | `When items are added, they will appear here.` |
| Error state | `Something went wrong. Check the details and try again.` |
| Destructive confirmation | `Delete item`: `This action cannot be undone. Delete this item?` |

Phase-specific copy rules:

- Shared `UiEmptyState` defaults to neutral generic copy only when a page does not provide more specific slot content.
- `UiConfirmDialog` must require explicit confirm copy from caller for destructive actions.
- Do not add visible instructional text that explains component mechanics.
- Keep Vietnamese/i18n strings where existing pages already use them. The shared UI kit must not hard-code English into feature surfaces except generic fallback examples in documentation.

Destructive actions in Phase 2:

- No new product destructive flows should be introduced.
- If a proof surface exposes existing delete/remove behavior, it must use `UiConfirmDialog` or preserve the existing confirmation behavior.

## Documentation Contract

Phase 2 implementation must leave downstream docs in or near the client UI kit location. Required content:

- Token rules: which `--st-*` tokens to use, which legacy tokens are compatibility-only.
- Tailwind-only rules: no SCSS, no scoped CSS, no component CSS, no broad global helper CSS.
- Component naming and API rules: `Ui*`, slot-first, small props.
- Variant tables for buttons, badges, alerts, inputs, cards, dialogs, skeletons.
- Migration map from old patterns to new primitives.

Required migration map entries:

| Old Pattern | New Primitive |
|-------------|---------------|
| `v-btn`, icon button | `UiButton`, `UiIconButton`, `AppIcon` |
| `v-text-field`, `v-textarea` | `UiInput`, `UiTextarea` |
| `v-select`, `v-autocomplete` | `UiSelect`; autocomplete deferred unless required by `/profile` later |
| `v-btn-toggle` | `UiSegmentedControl` |
| `v-card`, `v-sheet` | `UiCard`, `UiSection` |
| `v-table` | `UiTable` |
| `v-list`, `v-list-item` | `UiList`, `UiListItem` |
| `v-chip` | `UiBadge` |

| `v-alert` | `UiAlert` |
| `v-snackbar` / toast surface | `AppToast` using `UiToast` |
| `v-dialog` | `UiDialog`, `UiConfirmDialog` |

| `v-progress-linear` | `UiProgress` |
| `AppSkeleton` styled with CSS | `UiSkeleton` Tailwind-only |
| `TablePager` | `UiPagination` |
| `ClassLocation` chip/icon | `UiBadge` + `AppIcon` |

## Verification Contract

For Phase 2 code changes:

- Run `npm run lint` from `client/`.
- Run `npm run typecheck` from `client/`.
- Run `npm run build` from `client/`.
- If any proof surface changes visible UI, perform desktop and mobile-width visual QA and document evidence in the plan verification notes.
- Backend checks are not required unless backend files are touched.

Static scans expected after Phase 2:

- New `Ui*` components contain no `<style>`, `lang="scss"`, `.scss`, or Vuetify tags.
- New `Ui*` components contain no direct API/composable data fetching.
- No new dependency on Vuetify, Sass, shadcn, or third-party UI registries.
