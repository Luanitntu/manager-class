# Phase 03 - UI Design Contract

> Visual and interaction contract for Phase 3. Generated by gsd-ui-researcher for the planner, executor, checker, and auditor.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind 4 via `@tailwindcss/vite`; no shadcn; no Vuetify; no SCSS |
| Preset | Not applicable |
| Component library | Local Vue 3/Nuxt `Ui*` primitives from `client/app/components/ui` |
| Icon library | `@mdi/font` through `AppIcon`; do not introduce another icon library |
| Font | Nunito via `--st-font-family`; fallback `Arial, sans-serif` |
| Token source | Existing `--st-*` tokens in `client/app/assets/css/main.css` |
| Component prefix | `Ui*` for design-system primitives; `App*` remains app infrastructure |

Source: `03-CONTEXT.md`, `02-UI-SPEC.md`, `02-HANDOFF.md`, `client/app/components/ui/README.md`, current code scan.

## Phase Boundary

Phase 3 migrates app shell, auth, calendar, teacher/student dashboard, student schedule, and shared/high-traffic dialogs away from old Vuetify/CSS coupling. It is a migration and polish phase, not a priority page redesign.

In scope:

- Preserve role-aware app shell behavior for teacher, student, assistant, and admin branches.
- Keep teacher calendar access prominent in navigation and topbar.
- Migrate full auth surface: `AuthShell`, login, register, forgot password, reset password, verify email, and auth wrappers.
- Migrate teacher calendar workflow first: header, board, detail panel, feedback, drag/drop reschedule, create/edit dialog integration.
- Migrate teacher dashboard, then student dashboard and student schedule.
- Migrate `SessionDialog` with stronger QA; migrate low-risk portions of `StudentDetailDialog` and `AssistantDetailDialog` or defer complex tabs/forms with explicit inventory.
- Replace old snackbars with `AppToast`; replace old Vuetify controls with `Ui*` primitives.
- Leave touched Phase 3 surfaces scan-clean for `<v-*`, `</v-*`, `<style>`, `lang="scss"`, `.scss`, and `.v-*` coupling unless explicitly deferred.

Out of scope:

- Phase 4 priority page redesigns: `/assistants`, `/assistants/[id]`, `/audit-logs`, `/profile`.
- Admin-heavy and center-adjacent page redesigns unless shell/nav compatibility requires a minimal check.
- Center role workflows.
- New backend product behavior or new API contracts.
- Third-party UI registries or dependency installs.

## Tailwind-Only Migration Contract

All touched Phase 3 UI must use Tailwind utilities and local shared components. Do not add or retain component/page CSS in migrated surfaces.

Required replacements:

| Old pattern | Phase 3 replacement |
|-------------|---------------------|
| `v-btn`, icon buttons | `UiButton`, `UiIconButton`, `AppIcon` |
| `v-text-field`, `v-textarea` | `UiInput`, `UiTextarea` |
| `v-select`, basic autocomplete use | `UiSelect`; custom autocomplete remains page-owned if needed |
| `v-btn-toggle` | `UiSegmentedControl` |
| `v-card`, `v-sheet`, `v-container`, `v-row`, `v-col` | `UiCard`, `UiSection`, Tailwind grid/flex |
| `v-list`, `v-list-item` | `UiList`, `UiListItem`, Tailwind rows |
| `v-chip`, `v-chip-group` | `UiBadge`, native checkbox/toggle group, or explicit Tailwind buttons |
| `v-avatar`, avatar `v-img` | `UiAvatar` or `img` with Tailwind |
| `v-alert` | `UiAlert` |
| `v-snackbar` | `AppToast` using `UiToast` |
| `v-dialog` | `UiDialog`, `UiConfirmDialog` |
| `v-tabs`, `v-window` | `UiTabs` with caller-owned panels |
| `v-progress-linear`, `v-progress-circular` | `UiProgress`, `UiSpinner`, or `UiSkeleton` |
| `v-table` | `UiTable` |
| `v-icon` | `AppIcon` |

Do not move API calls, auth/session logic, validation, mutations, filtering, timezone conversion, tenant-aware behavior, or formatting into `Ui*` components. Existing composables and page/dialog ownership remain the data boundary.

## Spacing Scale

Declared values for all touched Phase 3 surfaces. All values are multiples of 4.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, badge gaps, dense metadata |
| sm | 8px | `gap-2`, `p-2` | Compact controls, row internals, inline actions |
| md | 16px | `gap-4`, `p-4` | Default control groups, list rows, dialog body groups |
| lg | 24px | `gap-6`, `p-6` | Cards, dashboard panels, auth form cards |
| xl | 32px | `gap-8`, `p-8` | Page headers, auth split padding, dashboard hero body |
| 2xl | 48px | `gap-12`, `py-12` | Major auth/landing-like separation only |
| 3xl | 64px | `gap-16`, `py-16` | Rare page-level spacing; avoid in dense app surfaces |

Exceptions:

- Minimum interactive target: 44px for buttons, nav links, dialog close controls, profile menu items, calendar cells that act as buttons, and icon-only controls.
- Compact 36px controls allowed only for pagination, dense toolbar controls, profile menu icon actions, calendar previous/next buttons, and small dialog secondary buttons.
- Shell topbar height remains 64px; drawer width remains 256px unless visual QA proves overflow on mobile.
- Dialog max width: `SessionDialog` 560px, `AssistantDetailDialog` 640px, `StudentDetailDialog` 720px. Use `UiDialog` sizes that preserve those widths.
- Dialog viewport bounds: `max-height: calc(100vh - 32px)` and internal body scroll.
- Cards use 8px radius. Legacy 12px/14px dashboard cards must normalize to 8px when migrated unless a specific parity note explains why not.
- No card-in-card layouts. Use sections, lists, tables, or unframed groups inside cards.
- App content wrapper remains `w-full px-4 py-6 sm:px-6 lg:px-8`; page content max is `--st-content-max` when a page container is appropriate.

## Typography

Touched Phase 3 surfaces use exactly these four sizes and two weights.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 14px | 600 | 1.5 | Form labels, nav group labels, table headers, metadata, button text |
| Body | 16px | 400 | 1.5 | Default copy, input text, dialog body, row text |
| Heading | 20px | 600 | 1.2 | Card headings, dialog titles, section titles |
| Page Title | 24px | 600 | 1.2 | App page titles, auth screen titles, dashboard primary headings |

Rules:

- Use weight 400 for content and weight 600 for emphasis/actions. Do not introduce weight 500, 700, or 800 in migrated Phase 3 surfaces.
- Letter spacing is `0`.
- Do not scale font size with viewport width.
- Avoid oversized dashboard/auth hero type in app surfaces; migrated dashboard hero headings use 24px, not 30px+.
- Long names, emails, class names, and session topics must use `min-w-0`, truncation, wrapping, or responsive stacking to prevent overlap.
- Preserve existing i18n strings and visible labels unless this spec defines replacement copy.

## Color

Use existing `--st-*` tokens as source of truth.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--st-bg-soft` `#f5f8ff` | Teacher app background, auth canvas, empty canvas, soft bands |
| Secondary (30%) | `--st-surface` `#ffffff` | Cards, dialogs, inputs, tables, drawer, topbar |
| Accent (10%) | `--st-accent` `#ff6b00` | Reserved elements listed below only |
| Destructive | `#ef4444` | Delete/remove/destructive confirmations only |

Accent reserved for:

- Calendar-first emphasis: current/urgent session markers and schedule attention states.
- Non-destructive warning/attention badges.
- Student streak/urgent learning markers already using orange.
- Auth/brand continuity accents where already present.

Do not use accent for all primary actions. Primary actions use `--st-primary` `#0d6efd`; hover/pressed uses `--st-primary-dark` `#075dee`.

Additional semantic colors:

| Token | Value | Usage |
|-------|-------|-------|
| Primary | `--st-primary` `#0d6efd` | Main CTAs, selected nav state, links, selected calendar modes |
| Primary Dark | `--st-primary-dark` `#075dee` | Primary hover/active |
| Secondary Brand | `--st-secondary` `#49beff` | Informational badges and calm highlights |
| Text | `--st-text` `#17233c` | Main text |
| Muted | `--st-muted` `#475569` | Secondary text, helper text |
| Border | `--st-border` `#dce5f1` | Borders, dividers, inputs |
| Focus | `--st-focus` | Focus ring; use Tailwind ring equivalent where practical |
| Success | `#10b981` | Saved, completed, verified states |
| Warning | `#f97316` | Non-destructive warnings |
| Error | `#ef4444` | Errors and destructive state |

Color rules:

- Teacher and student shells may have distinct content feel, but both must resolve to shared token semantics.
- Status cannot be color-only. Pair status dots/badges with text.
- Disabled state uses opacity plus semantic disabled affordance; contrast remains readable.
- Do not create a one-hue blue UI. Keep neutral surfaces, blue primary, orange accent, green success, red destructive.

## Surface Contracts

### App Shell

The shell must preserve:

- Role-aware nav groups for `SUPER_ADMIN`, `ASSISTANT`, teacher/default, and student shell.
- Teacher nav includes dashboard and calendar in the daily group; calendar remains visible without scrolling at desktop drawer width.
- Student nav includes dashboard, student calendar, classes, documents, assignments, grades, and tests.
- Mobile drawer opens/closes with the topbar button, closes on route change, and closes on backdrop click.
- Desktop drawer is open by default at `lg` and above.
- Topbar search, `LanguageSwitcher`, announcement banner, notifications affordance, profile/avatar menu, logout, and teacher calendar quick action remain present.
- Profile menu closes on outside click, route change, and Escape.

Visual contract:

- Drawer width 256px; topbar height 64px; main content starts below topbar and uses left padding at desktop.
- Nav rows are 44px minimum, have visible active state, icon + text, and text truncation.
- Teacher active nav uses primary blue with left indicator; student active nav uses soft blue fill.
- Announcement banner uses `UiAlert` or equivalent tokenized alert, not raw old classes.
- Replace one-off buttons with `UiIconButton`/`UiButton` where practical, but do not change routing behavior.

### Auth Surface

Migrate `AuthShell`, login/register panes, visual panes, forgot/reset/verify pages.

Visual contract:

- Split auth layout: visual/brand panel on desktop left, form on right; single-column form on mobile.
- Auth form max width 400px; page padding 32px desktop, 24px mobile.
- Brand mark uses `AppIcon` inside a 36px or 40px tokenized square, not `v-avatar`.
- Visual pane images remain real education/classroom imagery already present; do not replace with gradients-only or SVG illustrations.
- Auth form cards are un-nested surfaces; avoid card inside auth panel card.

Interaction contract:

- Preserve VeeValidate/Zod validation, `rememberMe`, redirects, toasts, disabled social buttons, and submit loading states.
- Use `UiInput`, `UiCheckbox`, `UiButton`, `UiAlert`, `UiSpinner`, and `AppIcon`.
- Forgot/reset/verify success/error states must be visible and keyboard accessible.
- Verify-email `verifying` state uses `UiSpinner` plus status text; `ok` and `error` states use semantic icon + heading + action.

### Teacher Calendar

Teacher calendar is the highest-priority workflow anchor.

Preserve:

- Month/week view toggle.
- Today, previous, next controls.
- Create from header and from in-month calendar cells.
- Keyboard create from calendar cells using Enter and Space.
- Open session from event.
- Drag/drop reschedule when `canEdit` is true.
- Loading calendar skeleton.
- Session detail panel and edit affordance.
- `SessionDialog` create/edit integration and saved reload.
- Error/success feedback through `AppToast`.

Visual contract:

- Calendar board remains a single primary surface with toolbar above a 7-column grid.
- Weekday header and cells must not clip at desktop or mobile widths.
- Calendar event rows keep stable height near 26px, support cancellation styling, and truncate safely.
- `+N sessions` overflow indicator remains visible after three events.
- Today marker and drop target states are visible without relying only on color.
- Detail panel uses tokenized border/surface, active marker colored by session class, empty state with icon + copy.

### Student Calendar And Schedule

Preserve:

- Week navigation: previous, title, next, today event from parent.
- Loading list skeleton.
- Empty weekly schedule state.
- Grouping by day and session cards.
- Online/offline location labels and action labels.

Visual contract:

- Timeline keeps left date rail on desktop; stacks safely on narrow mobile without clipping.
- Session cards use `UiCard`/Tailwind, not scoped CSS.
- Online session primary action uses primary blue; offline/location action uses secondary style.
- Progress and task states use `UiProgress`, `UiBadge`, and `AppIcon`.

### Dashboards

Teacher dashboard migration comes before student dashboard.

Teacher dashboard must preserve:

- Next-session hero with calendar CTA.
- Metric cards.
- Upcoming sessions list with class/link/tags/action.
- Mini month calendar.
- Action items panel.
- Empty upcoming-session state.

Student dashboard must preserve:

- Next-session hero with real image.
- Stat cards.
- Upcoming session cards with urgent state.
- Course cards with progress.
- Tasks panel and study stats panel.
- Loading skeleton behavior.

Visual contract:

- Use `UiPage`, `UiSection`, `UiCard`, `UiMetricCard`, `UiList`, `UiBadge`, `UiProgress`, `UiEmptyState`, `UiButton`, and `AppIcon`.
- Hero sections remain full surface bands, not nested cards. Use restrained education SaaS styling.
- Remove scoped CSS and old `.v-*` coupling; replace layout with Tailwind grids.
- Dashboard cards use 8px radius, stable min heights, and no nested-card styling.
- Mobile layout stacks all grids to one column by 760px without overlap.

### Shared Dialogs

`SessionDialog` is calendar-critical and must be migrated or planned separately with stronger QA.

`SessionDialog` must preserve:

- `modelValue`, `session`, `prefill`, `saved`, and `update:modelValue` contract.
- Single vs recurring create mode.
- Class select and class location display.
- Date/start/end inputs and recurring date range.
- Repeat weekday multi-select.
- Lesson topic input.
- Instructor select with teacher + assistants.
- Create, update, bulk create, delete, mark completed, reopen.
- Timezone conversion using `useUserTimezone`.
- Error display from `extractApiError`.
- Disabled/loading guards for saving/deleting/status updates.

Dialog visual/interaction contract:

- Use `UiDialog` size `md`/560px.
- Use `UiSegmentedControl` for single/recurring.
- Use `UiSelect`, `UiInput`, `UiBadge`, `UiAlert`, `UiButton`, `UiActionGroup`, `UiConfirmDialog` where applicable.
- Delete session is destructive and requires explicit confirmation before mutation.
- Footer order: destructive/status actions left, cancel and primary save right on desktop; stack actions on mobile.
- Preserve close behavior and focus restoration from `UiDialog`.
- Save button disabled until required class and relevant date/time inputs are present.

`StudentDetailDialog` low-risk migration scope:

- Header/avatar/close, tabs, profile fields, score form controls, score list, comments entry, empty states may migrate if kept behavior-compatible.
- Use `UiDialog` 720px, `UiTabs`, `UiInput`, `UiTextarea`, `UiSelect`, `UiList`, `UiIconButton`, `UiEmptyState`.
- Delete score is destructive and requires confirmation or an explicit deferred note if existing behavior is preserved temporarily.
- Complex timeline styling may be a simple list if comments remain readable and ordered.

`AssistantDetailDialog` low-risk migration scope:

- Header/avatar/close, salary config form, salary summary metrics, salary table, assigned class badges, empty states may migrate.
- Use `UiDialog` 640px, `UiInput`, `UiSelect`, `UiMetricCard`, `UiTable`, `UiBadge`, `UiSkeleton`.
- Save salary CTA label must be `Save salary settings`.

If a dialog area is not migrated, the plan must list exact remaining old markers and why deferral is safer than forced rewrite.

## Interaction Rules

Keyboard and focus:

- Every control, link, menu trigger, nav row, calendar cell, event button, tab, dialog close, pagination button, and confirm action is keyboard reachable.
- Focus-visible ring uses `--st-focus` or `focus-visible:ring-2 focus-visible:ring-blue-100`.
- Dialog opens move focus into the dialog and restore focus to the opener when closed.
- Escape closes non-destructive dialogs and profile menus. Destructive confirmations require explicit cancel/confirm.
- Icon-only controls require `aria-label`.

Responsive behavior:

- Shell drawer overlays below `lg`; no horizontal page scroll from drawer/topbar.
- Auth visual pane hides on mobile; form remains centered and readable.
- Toolbars wrap; actions stack on narrow widths.
- Tables use horizontal overflow wrappers below tablet width.
- Dialogs use `width: min(100vw - 32px, maxWidth)` and internal body scroll.
- Calendar grid must stay usable at mobile width; event labels may truncate but controls cannot overlap.

States:

- Every migrated data surface defines loading, empty, error, and populated states.
- Loading states preserve approximate layout dimensions with `AppSkeleton`/`UiSkeleton`.
- Mutating controls show loading and disabled state.
- Error states include problem text and a retry/next-step path when the source flow supports it.
- Success feedback uses `AppToast` or inline `UiAlert`, not blocking dialogs unless the existing workflow requires it.

Accessibility:

- Inputs have visible labels or accessible labels.
- Status dots and badges include text.
- Alerts/toasts use semantic roles.
- Color is never the only signal for status, selection, validation, active nav, today, drop target, or destructive action.

## Copywriting Contract

Preserve existing app/i18n copy in touched surfaces unless changing component implementation requires a direct replacement. Do not hard-code new English copy into Vietnamese/i18n surfaces when an existing key/string exists.

| Element | Copy |
|---------|------|
| Primary CTA | `Create session` for new calendar session, `Save session` for edited calendar session, `Save salary settings`, `Send reset link`, `Update password`, `Continue to sign in` |
| Empty state heading | Calendar/dashboard/schedule: `No upcoming sessions`; student schedule: `No classes this week`; scores: `No scores yet`; comments: `No comments yet`; assistant classes: `No class assignments yet` |
| Empty state body | Calendar/dashboard/schedule: `Create a session or adjust the selected week.`; scores: `Add a score after selecting a class.`; comments: `Add a comment to record student progress.`; assistant classes: `Assign this assistant from a class.` |
| Error state | `Something went wrong. Check the details and try again.` Use `extractApiError` output when available. |
| Destructive confirmation | `Delete session`: `This action cannot be undone. Delete this session?`; `Delete score`: `This action cannot be undone. Delete this score?` |

Copy rules:

- CTA labels use specific verb + noun, not generic `Save` when the object is ambiguous.
- Primary, save, and confirm actions in touched Phase 3 surfaces must use object-specific visible labels/i18n values such as `Save session`, `Save salary settings`, `Delete session`, and `Delete score`; do not use `common.save` or generic `Save` / `Cancel` as CTA labels.
- `common.cancel` may be used only for secondary non-CTA dismissal controls.
- Existing i18n keys such as `session.newSession` and `session.editSession` may remain if current localization already supplies the visible copy.
- Do not add visible instructional text explaining UI mechanics or keyboard shortcuts.
- Disabled social auth buttons keep existing unavailable titles or equivalent accessible labels.

Destructive actions in Phase 3:

- `SessionDialog` delete session: use `UiConfirmDialog`, destructive tone, explicit confirm/cancel.
- `StudentDetailDialog` delete score: use `UiConfirmDialog` if migrated; otherwise document as deferred old behavior.
- No other destructive actions should be introduced.

## Visual QA Contract

Required desktop and mobile-width QA for touched surfaces:

- Shell: teacher, student, assistant branches; admin branch only if shared shell changes affect it.
- Auth: login, register, forgot password, reset password, verify email.
- Teacher calendar: loading, populated month/week, create from cell, edit existing session, drag/drop result feedback, error toast path.
- Teacher dashboard: next-session present, no-next-session empty path, metric/list/card layout.
- Student dashboard and schedule: loading, populated, empty week, mobile stacking.
- Dialogs: `SessionDialog` create single, create recurring, edit, delete confirmation, mark done/reopen; migrated student/assistant dialog areas if included.

Pass criteria:

- No overflow, overlap, clipped controls, unreadable text, broken wrapping, missing focus, or missing loading/empty/error state.
- Mobile drawer and dialogs do not create horizontal scroll.
- Calendar-first teacher entry remains visible and usable.
- Role-aware nav routes remain present and clickable.
- Touched files are scan-clean for old UI markers unless explicitly deferred.

## Verification Contract

For Phase 3 frontend changes:

- Run `npm run lint` from `client/`.
- Run `npm run typecheck` from `client/`.
- Run `npm run build` from `client/`.
- Run static scan over touched Phase 3 surfaces for `<v-*`, `</v-*`, `<style>`, `lang="scss"`, `.scss`, `.v-*`, and `:deep(.v-*)`.
- Run static scan over `client/app/components/ui` to ensure no API/composable/data-fetching imports were introduced.
- Backend checks are not required unless backend files are touched.

If a command cannot run in the environment, record the exact reason and closest verification performed.
