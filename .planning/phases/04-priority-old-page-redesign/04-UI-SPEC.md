# Phase 04 - UI Design Contract

> Visual and interaction contract for Phase 4. Generated by gsd-ui-researcher for the planner, executor, checker, and auditor.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind 4 via `@tailwindcss/vite`; no shadcn; no Vuetify; no SCSS |
| Preset | Not applicable |
| Component library | Local Vue 3/Nuxt `Ui*` primitives from `client/app/components/ui`; `App*` infrastructure wrappers where already used |
| Icon library | `@mdi/font` through `AppIcon`; do not introduce another icon library |
| Font | Nunito via `--st-font-family`; fallback `Arial, sans-serif` |
| Token source | Existing `--st-*` tokens in `client/app/assets/css/main.css` |
| Component prefix | `Ui*` for design-system primitives; `App*` remains app infrastructure |

Source: `04-CONTEXT.md`, `02-UI-SPEC.md`, `02-HANDOFF.md`, `03-UI-SPEC.md`, `03-MIGRATION-INVENTORY.md`, `client/app/components/ui/README.md`, and current target source scan.

## Phase Boundary

Phase 4 redesigns only these priority old UI routes with the shared Tailwind UI kit while preserving existing behavior:

- `/assistants`
- `/assistants/[id]`
- `/audit-logs`
- `/profile`

In scope:

- Replace old Vuetify markup in the four route files with Tailwind/shared UI primitives.
- Preserve existing feature composables, query refs, mutation calls, pagination refs, filter refs, watchers, timezone formatting, auth-store updates, and i18n strings where present.
- Preserve loading, empty, error, success, populated, disabled, and saving states.
- Make the four routes visually consistent with the Phase 3 shell and Phase 2 UI kit at desktop and mobile widths.
- Leave the four route files scan-clean for `<v-*`, `</v-*`, `<style>`, `lang="scss"`, `.scss`, and `:deep(.v-)`.

Out of scope:

- Backend changes.
- New dependencies or third-party UI registries.
- Broad app cleanup outside the four named route files.
- Admin/center pages, general page migration, landing cleanup, stale style cleanup.
- New searchable timezone select, audit filter drawer, assistant card grid, or mobile stacked-card table rewrites.

## Tailwind-Only Route Contract

All touched Phase 4 route UI must use Tailwind utilities and local shared components. Do not add component CSS, page CSS, SCSS, CSS modules, Vuetify wrappers, or `v-*` compatibility shims.

Required old-to-new replacements:

| Old pattern | Phase 4 replacement |
|-------------|---------------------|
| `v-btn`, text/icon buttons | `UiButton`, `UiIconButton`, `AppIcon` |
| `v-text-field` | `UiInput` |
| `v-select`, `v-autocomplete` | `UiSelect`; no custom searchable select |
| `v-card`, `v-row`, `v-col`, `v-divider` | `UiCard`, Tailwind grid/flex/border utilities |
| `v-list`, `v-list-item` | `UiList`, `UiListItem`, or Tailwind rows when the row must be a `NuxtLink`/button |
| `v-chip` | `UiBadge` |
| `v-avatar`, `v-img` avatar usage | `UiAvatar` |
| `v-alert` | `UiAlert` |
| `v-dialog` | `UiDialog` |
| `v-tabs`, `v-window` | `UiTabs` with caller-owned panels |
| `v-table` | `UiTable` |
| `v-icon` | `AppIcon` |
| Existing pagination wrapper | `TablePager` may remain; direct `UiPagination` is allowed only if behavior stays identical |
| Existing skeleton wrapper | `AppSkeleton` may remain; direct `UiSkeleton` is allowed only if dimensions stay comparable |

Data ownership rule:

- Route pages keep `useAssistants`, `useAssistantMutations`, `useAssistantDetail`, `useAssistantSalarySummary`, `useAssistantSessions`, `useAuditLogs`, `useMyProfile`, `useProfileMutations`, `useAuthStore`, `useAvatar`, date/time helpers, `extractApiError`, and i18n ownership.
- `Ui*` components must remain data-agnostic and must not import feature composables, auth stores, API helpers, `$fetch`, or tenant-aware logic.

## Spacing Scale

Declared values for all touched Phase 4 surfaces. All values are multiples of 4.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, badge gaps, dense table metadata |
| sm | 8px | `gap-2`, `p-2` | Compact row internals, inline controls, toolbar action gaps |
| md | 16px | `gap-4`, `p-4` | Default card/list/table/form group spacing |
| lg | 24px | `gap-6`, `p-6` | Page sections, dialog/card padding, major vertical groups |
| xl | 32px | `gap-8`, `p-8` | Page header separation and rare route-level grouping |
| 2xl | 48px | `gap-12`, `py-12` | Large empty canvas only; avoid in dense route bodies |
| 3xl | 64px | `gap-16`, `py-16` | Not used in Phase 4 operational pages |

Exceptions:

- Minimum interactive target is 44px for primary buttons, row buttons/links, dialog close controls, pagination buttons, and profile save controls.
- Compact 36px controls are allowed only in dense filter toolbars, pagination, and small secondary dialog actions.
- Create assistant dialog max width remains about 460px; use `UiDialog size="sm"` with body content constrained if needed.
- Assistant profile edit dialog remains compact at about 440px; use `UiDialog size="sm"`.
- Profile form card max width remains 520px.
- Tables use stable 48px minimum row height and horizontal scroll at mobile width.
- Assistant list rows use 56px minimum height with 16px row padding and stable avatar alignment.
- Cards use `--st-radius` 8px. Do not put cards inside cards.
- Route pages use `UiPage` default width unless `/profile` uses a narrow internal max width.

## Typography

Touched Phase 4 surfaces use exactly these four sizes and two weights.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 14px | 600 | 1.5 | Form labels, table headers, metadata, badge/button text |
| Body | 16px | 400 | 1.5 | Default row text, input text, table cells, page copy |
| Heading | 20px | 600 | 1.2 | Card headings, dialog titles, section titles, metric values |
| Page Title | 24px | 600 | 1.2 | Route page titles only |

Rules:

- Use weight 400 for content and 600 for emphasis/actions. Do not introduce weights 500, 700, or 800 in touched Phase 4 route markup.
- Letter spacing is `0`.
- Do not scale font size with viewport width.
- Long names, emails, phones, class names, entity IDs, and action strings must use `min-w-0`, wrapping, truncation, or horizontal table scroll to prevent overlap.
- Table body text may use 14px when using `UiTable`'s existing `text-sm` contract; page body/form/list text remains 16px.
- Preserve existing i18n copy where present unless this spec defines a direct replacement.

## Color

Use existing `--st-*` tokens as source of truth.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--st-bg-soft` `#f5f8ff` | App/page background inherited from shell; soft empty/state accents |
| Secondary (30%) | `--st-surface` `#ffffff` | Cards, lists, tables, dialogs, inputs |
| Accent (10%) | `--st-accent` `#ff6b00` | Reserved elements listed below only |
| Destructive | `#ef4444` | Error alerts, deleted audit action badges, destructive confirmations only |

Accent reserved for:

- Non-destructive warning/attention badges only.
- Existing brand/auth continuity outside these route bodies.
- Calendar-first highlights outside Phase 4 route bodies.

Do not use accent for default primary actions. Primary actions use `--st-primary` `#0d6efd`; hover/pressed uses `--st-primary-dark` `#075dee`.

Additional semantic colors:

| Token | Value | Usage |
|-------|-------|-------|
| Primary | `--st-primary` `#0d6efd` | Main CTAs, links, selected tabs, primary badges |
| Primary Dark | `--st-primary-dark` `#075dee` | Primary hover/active |
| Secondary Brand | `--st-secondary` `#49beff` | Informational badges and calm highlights |
| Text | `--st-text` `#17233c` | Main text |
| Muted | `--st-muted` `#475569` | Secondary text, helper text, metadata |
| Border | `--st-border` `#dce5f1` | Borders, dividers, inputs, table/list separators |
| Focus | `--st-focus` | Focus ring via existing Tailwind ring equivalents |
| Success | `#10b981` | Successful save, created audit actions, positive salary emphasis |
| Warning | `#f97316` | Non-destructive warnings only |
| Error | `#ef4444` | Error alerts and deleted audit actions |

Badge tone mapping:

- Audit action `CREATED` -> `UiBadge tone="success"`.
- Audit action `UPDATED` -> `UiBadge tone="info"`.
- Audit action `DELETED` -> `UiBadge tone="danger"`.
- Audit action `RECORDED` -> `UiBadge tone="primary"` or `info`.
- Audit fallback -> `UiBadge tone="neutral"`.
- Assistant class count -> `neutral`.
- Assistant salary method/rate -> `primary` or `info`.
- Assistant level -> `info`.
- Session `SCHEDULED` -> `info`, `COMPLETED` -> `success`, `CANCELLED` -> `danger`.
- Assigned whole-class role -> `primary`; teaches-only/session role -> `info`.

Color rules:

- Status cannot be color-only; badges must render text.
- Disabled state uses opacity plus disabled attribute.
- Focus-visible state must remain visible on every keyboard-focusable control.
- Do not create a one-hue blue UI; use neutral surfaces, blue primary, green success, red error, and limited orange attention.

## Route Surface Contracts

### `/assistants`

Visual structure:

- Use `UiPage` and `UiPageHeader`.
- Header title: `Assistant Teachers`; subtitle: `Assignments and salary.`; primary action `Create assistant` with `mdi-account-plus`.
- Search sits below header in a wrapping `UiToolbar`, max width about 360px on desktop and full width on mobile.
- Main list uses one white `UiCard` or `UiList` surface. Do not use a card grid.
- Assistant rows are dense list-row/lightweight-card style: avatar left, name/email/phone center, salary and class-count badges right on desktop, badges wrap under row text on mobile.
- Empty state stays compact inside the list/card area with `UiEmptyState`, not a full onboarding panel or illustration.

Interaction contract:

- Preserve `search`, `page`, `limit`, `useAssistants(search, page, limit)`, and `watch([search, limit], () => page.value = 1)`.
- Preserve `openDetail(id)` behavior and `AssistantDetailDialog v-model` with `selectedId`.
- Preserve `TablePager`/`UiPagination` behavior and limit/page refs.
- Create dialog stays compact around 460px with `UiDialog`.
- Preserve create form fields: full name, email, temporary password, phone optional.
- Preserve disabled rule: create is disabled when full name or email is missing, or password length is less than 8.
- Preserve create loading guard and `extractApiError(e) ?? 'Could not create assistant'`.
- On successful create, close dialog and reset form to empty values.

States:

- Loading with no rows: `AppSkeleton`/`UiSkeleton variant="list"` with 5 rows inside the list area.
- Empty: heading `No assistants found`; body `Create an assistant or change your search.`; action `Create assistant`.
- Error: inline `UiAlert tone="error"` inside dialog for create errors.

### `/assistants/[id]`

Visual structure:

- Use `UiPage`.
- Top back action links to `/assistants` using `UiButton variant="ghost"` with `mdi-arrow-left`.
- Header uses avatar, assistant name, email, optional level badge, and edit profile action. Stack on mobile.
- Desktop summary uses two columns: profile card and salary summary card side by side; stack below tablet width.
- Assigned classes, salary configuration, and schedule/breakdown/history are full-width sections below.
- Profile details use compact icon rows, not nested cards.
- Salary summary uses `UiMetricCard` grid for this month, total salary, sessions, hours, and next payroll.
- Assigned classes use a row/list surface; class rows may link to `/classes/:id`; badges show role/session label.
- Salary configuration remains an inline card with method/rate/effective date inputs and save action on the page.
- Rate history remains visible below salary config when more than one rate exists.
- Schedule, salary breakdown, and salary history remain table-based with horizontal overflow on mobile.
- Use `UiTabs` for `schedule`, `breakdown`, and `history`; caller owns panel rendering.
- Edit profile remains a compact `UiDialog` for phone, level, and hometown.

Interaction contract:

- Preserve `route.params.id`, `useAssistantDetail`, `useAssistantSalarySummary`, `useAssistantSessions`, `useAssistantMutations`, `useAvatar`, `useUserTimezone`, `useI18n`.
- Preserve computed involvement merge of class assignments and sessions.
- Preserve `tab = 'schedule'` default.
- Preserve `openProfile()` prefill from current assistant data.
- Preserve salary form watcher from assistant profile with `salaryMethod`, `salaryRate`, `effectiveFrom`.
- Preserve `payload(extra)` merging profile and salary fields.
- Preserve `saveSalary()` calling `updateSalary.mutateAsync({ id, body: payload({}) })`.
- Preserve `saveProfile()` calling update salary endpoint with profile fields, then closing the dialog.
- Preserve date/time formatting helpers: `money`, `dayOf`, `timeRange`, `fmtDate`, `fmtMonth`.
- Save salary CTA label must be `Save salary settings`, not generic `Save`.
- Save profile CTA label should be `Save profile`.

States:

- Initial loading: `AppSkeleton`/`UiSkeleton variant="detail"` before assistant data is present.
- Assigned classes empty: compact `UiEmptyState` heading `No class assignments yet`; body `Assign this assistant from a class.`
- Schedule empty table copy: `No sessions scheduled`.
- Breakdown empty table copy: `No salary records yet`.
- History empty table copy: `No salary history yet`.
- Mutating salary/profile controls show loading and disabled state from `updateSalary.isPending.value`.
- No destructive actions are introduced.

### `/audit-logs`

Visual structure:

- Use `UiPage` and `UiPageHeader`.
- Header title: `Audit Logs`; subtitle: `Important actions performed in your tenant.`
- Filters use horizontal wrapping `UiToolbar`; do not add collapsed filters or drawer.
- Toolbar fields: action select, entity type input, from date, to date, clear filters action.
- Action/entity/from/to controls stay visible together on desktop and wrap cleanly on mobile.
- Main data surface uses `UiCard`/`UiTable` with horizontal overflow on mobile.
- Preserve columns: When, Actor, Action, Entity.
- Use compact table skeleton, not page-wide loading hero.
- Empty state stays compact inside table/card area.

Interaction contract:

- Preserve `action`, `entityType`, `from`, `to`, `page`, `limit`.
- Preserve `watch([action, entityType, from, to, limit], () => page.value = 1)`.
- Preserve `useAuditLogs({ action, entityType, from, to, page }, limit)`.
- Preserve `clearFilters()` setting action, entity type, from, and to to empty strings.
- Preserve date formatting with `new Date(iso).toLocaleString()`.
- Preserve entity display: entity type plus first 8 characters of entity id when present.
- Preserve actor display: actor full name fallback dash plus role in parentheses when present.
- Preserve `TablePager`/`UiPagination` behavior with `meta`.

States:

- Loading with no logs: `AppSkeleton`/`UiSkeleton variant="table"` with 6 rows and 4 columns.
- Empty: heading `No audit logs found`; body `Adjust filters to view matching activity.`
- Action badges follow the semantic mapping in Color.
- No destructive actions are present.

### `/profile`

Visual structure:

- Use `UiPage`.
- Header title: `Profile`; subtitle: `Manage your account details.`
- Body is a compact single `UiCard`, max width 520px. Do not expand to a two-column account page.
- Loading uses `AppSkeleton`/`UiSkeleton variant="form"` with the same 520px max width.
- Email is a disabled `UiInput`.
- Full name and phone use `UiInput`.
- Timezone uses simple native `UiSelect` over the existing timezone list. Do not build a searchable select.
- Role uses `UiBadge`.
- Save feedback stays inline in the card with `UiAlert`, not toast.

Interaction contract:

- Preserve `useAuthStore`, `useMyProfile`, `useProfileMutations`, `detectBrowserTimezone`, and `timezones` fallback logic.
- Preserve watcher that copies loaded profile into `form.fullName`, `form.phone`, `form.timezone` and updates `auth.user.avatarKey`.
- Preserve save guard when `loading.value` is true.
- Preserve save mutation body: fullName, phone optional, timezone optional.
- Preserve successful auth store updates: `auth.user.fullName` and `auth.user.timezone`.
- Preserve `saved = true` success state and `extractApiError(e) ?? 'Could not save profile'` error state.
- Primary CTA label must be `Save profile`.

States:

- Loading: form skeleton.
- Error alert: `UiAlert tone="error"` with error text.
- Success alert: `UiAlert tone="success"` copy `Profile updated.`
- Disabled save/loading state during mutation.
- No destructive actions are present.

## Interaction Rules

Keyboard and focus:

- Every input, select, button, row button/link, tab, dialog close, pagination control, and clear-filter action is keyboard reachable.
- Focus-visible ring uses `--st-focus` or existing `focus-visible:ring-2 focus-visible:ring-blue-100`.
- `UiDialog` opens move focus into the dialog and restore focus to the opener when closed.
- Escape closes create/edit dialogs.
- Icon-only controls require `aria-label`.

Responsive behavior:

- Toolbars wrap instead of overflowing.
- Assistant rows stack metadata/badges under the main text on narrow screens.
- Tables use horizontal overflow wrappers on mobile; do not convert Phase 4 tables to bespoke stacked cards.
- Dialogs use `width: min(100vw - 32px, maxWidth)` through `UiDialog` and internal body scroll.
- Profile card remains readable at mobile width with full-width controls.
- No route may create horizontal page scroll except inside intentional table overflow containers.

State behavior:

- Every touched route defines loading, empty, error where applicable, and populated states.
- Loading states preserve approximate layout dimensions with skeletons.
- Mutating controls show loading and disabled state.
- Error states include problem text and a next-step path when available.
- Pagination range math and page-size behavior remain compatible with `TablePager`/`UiPagination`.

Accessibility:

- Inputs/selects have visible labels or accessible labels.
- Tables have captions via `UiTable caption` or equivalent screen-reader text.
- Status badges include text.
- Alerts use semantic roles from `UiAlert`.
- Color is never the only signal for status, selection, validation, or disabled state.
